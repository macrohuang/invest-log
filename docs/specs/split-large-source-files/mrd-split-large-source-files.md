# 大文件拆分重构 MRD

## 1. 背景与目标
- 背景问题：项目中存在多个超大源码文件，前端与 Go 后端职责混杂、可读性下降，后续修改和回归成本较高。
- 业务目标：在 **不改变功能、不改变对外签名/调用方式** 的前提下，对目标文件做结构化拆分与整理，使其更符合前端与 Go 工程规范。
- 成功指标：
  - 目标文件功能、路由、接口字段、调用签名保持一致。
  - 重构后职责边界更清晰，主要逻辑拆分到同目录的新模块文件。
  - 相关测试通过，且变更涉及的 Go 包覆盖率维持在 `>=80%`。

## 2. 用户与场景
- 目标用户：本仓库维护者与后续开发者。
- 核心场景：开发者需要在设置页、AI 聊天客户端、价格抓取、外部数据抓取等模块上继续迭代，希望降低单文件复杂度并提高可维护性。
- 非目标用户：终端使用者不会看到交互或功能层面的变化。

## 3. 范围定义
- In Scope：
  - `static/modules/pages/settings.js`
  - `go-backend/pkg/investlog/ai_chat_client.go`
  - `internal/handlers/handlers.go`
  - `go-backend/pkg/investlog/price_fetcher.go`
  - `go-backend/pkg/investlog/ai_symbol_data_fetcher.go`
  - 与上述拆分直接相关的同目录新增文件、必要的导入调整、必要测试补充/修正。
- Out of Scope：
  - API 行为、数据库结构、业务规则变化。
  - 非直接受拆分影响的功能优化或缺陷修复。
  - 未在本次范围内的其他大文件（例如若用户本意是 `go-backend/internal/api/handlers.go`，需单独确认后处理）。

## 4. 需求明细
- 功能需求 FR-1：保持对外行为完全一致，包括 HTTP 行为、返回结构、前端页面交互与数据提交流程。
- 功能需求 FR-2：保持现有可见函数/方法签名一致；如拆分到新文件，仍保留原 package/module 的可调用方式。
- 功能需求 FR-3：按职责拆分实现，例如常量、类型、请求执行、解析、UI 片段、事件绑定、数据转换等分离。
- 非功能需求 NFR-1：单文件行数应明显下降，避免继续维持超大“万能文件”。
- 非功能需求 NFR-2：遵循现有仓库规范：Go 使用 `gofmt`、小文件按关注点拆分；前端保持 2 空格缩进、函数聚焦、早返回。

## 5. 约束与依赖
- 技术约束：
  - Go 侧必须遵守现有 package 结构，不引入破坏性包边界变更。
  - 前端侧保持现有模块入口与调用点，避免改动路由或页面装配方式。
- 兼容性约束：
  - 不修改外部 API 契约。
  - 不修改已有数据格式与持久化结构。
- 依赖：
  - 依赖现有测试覆盖作为回归保障。

## 6. 边界与异常
- 边界条件：若某文件本身已很小（如 `internal/handlers/handlers.go` 当前仅几十行），则仅做最小必要整理，不强行拆分。
- 边界条件：若完全达成统一“目标行数”会显著破坏可读性，则优先保证职责清晰和行为零变化。
- 异常处理：若在重构过程中发现原实现存在隐式耦合，允许通过新增私有 helper 文件消化，但不能修改外部行为。

## 7. 验收标准
- AC-1：目标模块现有测试全部通过。
- AC-2：变更相关 Go 包测试通过且覆盖率 `>=80%`。
- AC-3：目标文件及其拆分后的新文件职责清晰，不再把多个独立 concern 长期堆在同一文件。
- AC-4：对外函数/方法签名、页面入口、HTTP handler 路由行为保持不变。

## 8. 待确认问题
- Q1：用户列出的 `internal/handlers/handlers.go` 是否为精确范围；本次默认按字面路径处理，不自动替换成 `go-backend/internal/api/handlers.go`。
- Q2："文件 tfdc 行数不能太多" 解释为“单文件行数不宜过高”；本次默认按“尽量显著下降并控制在合理范围”执行，不承诺机械固定阈值。

## 9. 假设与决策记录
- 假设：本次目标是纯重构，不引入任何功能变化。
- 假设：可以新增同目录文件来承接拆分后的实现。
- 决策：优先采用“按职责拆分文件 + 保持原入口薄封装”的方式，降低风险并保留调用兼容性。
