# 个股 AI 多框架分析 MRD

## 1. 背景与目标
- 背景问题：当前个股分析固定为宏观/行业/公司/国际四维，缺少“财报时间窗 + 框架化估值/质量/竞争分析”的强约束，输出风格与概率表达也不够硬。
- 业务目标：将个股 AI 分析升级为“财报优先 + 宏观产业实时信息 + 3框架深度分析 + 综合建议”，输出可直接执行。
- 成功指标（可量化）：
  - 100% 输出包含 3 个框架分析结果。
  - 100% 输出包含顶部综合建议（首句直接给结论）。
  - 100% 输出 `action_probability_percent` 为 0-100 明确数值。
  - 100% 输出 `disclaimer` 长度 <= 16 字。
  - 100% 输出不包含“看情况/视情况而定/it depends”。

## 2. 用户与场景
- 目标用户：已有持仓用户，使用个股 AI 分析接口评估加仓/持有/减仓。
- 核心使用场景：用户选择单一标的，系统拉取近 5 季报 + 近 3 年报 + 外部行业/政策/经营动态，生成 3 框架分析与综合建议。
- 非目标用户：无资产配置或无持仓信息的纯行情查询用户。

## 3. 范围定义
- In Scope：
  - 在 `go-backend/pkg/investlog/ai_symbol_analysis.go` 中调整个股分析流程与提示词约束。
  - 强制纳入数据抓取优先级：近 5 季报、近 3 年报、行业宏观政策、产业周期、公司最新经营。
  - 从指定框架池中选择 3 个框架开展分析，并返回每个框架结果。
  - 在综合建议中纳入“持仓数量 + 仓位占比 + 资产类别配置区间 + 用户偏好与策略”权重。
  - 输出语气与结构硬约束（概率、首句结论、短 disclaimer、禁含混表达）。
  - 补齐单元测试与回归测试。
- Out of Scope：
  - 新增前端页面/字段展示逻辑。
  - 新增数据库表结构迁移（优先复用现有 `symbol_analyses` 结构）。
  - 新增外部商业数据源账号体系。

## 4. 需求明细
- 功能需求 FR-1：系统必须先抓取并组织以下数据上下文再进入框架分析：近 5 个季度财报、近 3 年年报、行业宏观政策、产业周期、公司最新经营动态。
- 功能需求 FR-2：系统必须从指定框架池中选取 3 个框架并逐一输出分析结论与建议（每框架可审计）。
- 功能需求 FR-3：系统必须输出顶部综合建议，且综合评级需包含冲突权衡逻辑与可执行行动项。
- 功能需求 FR-4：系统必须输出明确概率（数值），且禁止含混措辞。
- 功能需求 FR-5：综合建议权重计算必须纳入持仓数量、仓位占比、资产类别配置区间、用户偏好与策略。
- 功能需求 FR-6：`disclaimer` 字段必须为短风险锚点（<=16 字）。
- 非功能需求 NFR-1：保持现有接口兼容（请求体不扩展为必填；返回字段向后兼容），失败场景可降级但不能崩溃。

## 5. 约束与依赖
- 技术约束：
  - 继续基于现有 AI 调用链路（`aiChatCompletion`）。
  - 保持 Go 代码风格与现有并发模式。
- 数据约束：
  - 外部数据可能缺失或超时，需可降级。
  - 财报字段可能在不同市场格式不一致，需在提示词中容错。
- 外部依赖：
  - 现有 `ai_symbol_data_fetcher.go` 的外部源。
  - 大模型对结构化 JSON 的遵循能力。

## 6. 边界与异常
- 边界条件：
  - 无持仓标的也允许分析，但权重字段需标注未知/默认值。
  - 无配置区间时默认区间处理需明确。
- 异常处理：
  - 外部数据抓取失败时降级到已有上下文继续分析。
  - 单个框架解析失败不应导致全流程崩溃，但最低成功框架数要有门槛。
- 失败回退：
  - 达不到最小可用框架数时返回失败并记录错误。

## 7. 验收标准
- AC-1（可验证）：分析结果中包含且仅包含 3 个框架分析对象，并有顶部综合建议。
- AC-2（可验证）：综合建议首句直接结论，且包含明确概率数值。
- AC-3（可验证）：输出中不出现“看情况/视情况而定/it depends”。
- AC-4（可验证）：`disclaimer` 长度 <= 16 字。
- AC-5（可验证）：综合建议文本体现持仓与配置区间差值、权重与冲突权衡逻辑。

## 8. 待确认问题
- Q1：框架选择策略是否固定为“与标的最相关 3 个”，还是允许随机探索并在结果中解释？
- Q2：当财报数据不全时，是否允许用 TTM/管理层指引替代并继续输出评级？

## 9. 假设与决策记录
- 假设：
  - 现有请求参数已足够，不新增 API 必填参数。
  - 现有外部数据抓取能力可提供至少部分财务/经营上下文。
  - 若无法稳定结构化存储“每框架结果”，优先放入现有维度字段或综合字段文本中，确保接口可用。
- 决策：
  - 先做后端提示词与解析/归一化增强，确保行为达标。
  - 优先通过单测约束“概率、措辞、免责声明、首句结论、权重因子”。
