# MRD

> 文件名：`docs/specs/log-json-newline/mrd-log-json-newline.md`

## 1. 背景与目标
- 背景问题：AI 请求日志此前为了便于阅读，采用 Pretty JSON 输出；在控制台日志中这会产生大量真实换行，导致单条日志被拆散，阅读和检索都很差。
- 业务目标：保留请求日志里的结构化信息与敏感字段脱敏能力，同时去掉额外格式化换行，让控制台输出保持紧凑。
- 成功指标（可量化）：`ai request json` 日志输出为单行 JSON 字符串，不再包含 Pretty Print 产生的真实换行。

## 2. 用户与场景
- 目标用户：开发者、排查线上 AI 请求问题的维护人员。
- 核心使用场景：在终端、日志聚合平台或本地文件中查看 `ai request json` 日志时，快速定位完整请求信息。
- 非目标用户：不直接查看后端日志的普通终端用户。

## 3. 范围定义
- In Scope：调整 AI 请求日志 JSON 编码方式；补充对应单元测试与交付文档。
- Out of Scope：改动其他日志类别；引入新的日志格式配置；重新设计控制台渲染样式。

## 4. 需求明细
- 功能需求 FR-1：`formatAIRequestForLog` 生成的日志 JSON 需为单行紧凑格式。
- 功能需求 FR-2：Header 脱敏规则、方法、URL、请求体字段信息保持不变。
- 非功能需求 NFR-1（性能/安全/可靠性等）：日志输出仍需是有效 JSON 字符串，不能泄露完整密钥或 Token。

## 5. 约束与依赖
- 技术约束：继续使用 Go 标准库 `encoding/json`；不改变现有 `slog` 调用方式。
- 数据约束：请求体可能是 JSON，也可能是非 JSON 原始文本。
- 外部依赖：无。

## 6. 边界与异常
- 边界条件：空 body 不应写入 body 字段；多行 JSON body 也应被压缩到单行日志表示。
- 异常处理：若日志 payload 编码失败，返回紧凑错误 JSON。
- 失败回退：维持当前错误信息内容，不中断正常业务请求。

## 7. 验收标准
- AC-1（可验证）：JSON 请求体日志输出中不含 Pretty Print 产生的真实换行。
- AC-2（可验证）：非 JSON 原始 body 仍能记录，且输出中敏感 Header 继续被脱敏。

## 8. 待确认问题
- Q1：无，按用户明确要求移除控制台中多余换行。
- Q2：无。

## 9. 假设与决策记录
- 假设：用户提到的“多余的 `\n`”主要指 Pretty JSON 输出造成的真实换行，而不是字符串内容本身的转义换行。
- 决策：采用紧凑 JSON 编码替代 `json.MarshalIndent`，不新增配置开关。
