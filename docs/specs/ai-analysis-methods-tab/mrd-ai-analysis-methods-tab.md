# MRD：AI Analysis 顶级 Tab 与可配置分析方法

## 1. 背景
- 当前系统已有 AI 相关能力，但入口分散在持仓分析、标的分析和 Settings 中。
- AI 分析流程是固定形态，用户无法在产品内配置多种分析方法，也无法复用不同 Prompt 模板。
- 现有 Settings 仅保存一套 AI 连接参数和若干偏好项，不支持对“分析方法”进行管理。

## 2. 目标
- 在顶级导航中新增 `AI Analysis` Tab，提供一个通用 AI 分析工作台。
- 用户进入后可以选择一个分析方法，系统根据 Prompt 中的 `${VAR}` 自动生成输入项。
- 分析方法的 `Name`、`System Prompt`、`User Prompt` 在 Settings 中可新增、编辑、删除。
- 运行分析时支持流式输出，并将每次执行结果持久化，供后续查看历史。

## 3. 用户与核心场景
- 目标用户：使用 Invest Log 管理投资组合，并希望复用自定义 Prompt 模板进行研究分析的个人投资者。
- 场景 A：用户在 Settings 中新增“估值复盘”“财报拆解”“行业对比”等分析方法。
- 场景 B：用户进入 `AI Analysis`，选择某个方法，填写变量后运行，实时看到 AI 输出。
- 场景 C：用户回看某个方法的历史执行记录，了解当时输入变量和输出内容。

## 4. 范围
- In Scope
  - 顶级导航新增 `AI Analysis` 页面。
  - Settings 支持分析方法的增、删、改。
  - 根据 `${VAR}` 自动提取变量并生成表单。
  - 运行分析时仅使用用户填写变量替换占位符，不自动注入持仓/交易等系统上下文。
  - 支持 SSE 流式输出。
  - 持久化分析历史，包括方法快照、变量输入、渲染后的 Prompt、结果和状态。
- Out of Scope
  - 变量类型元数据配置（如数字、下拉、日期、必填说明）。
  - 方法排序、分组、收藏、启停。
  - 自动从持仓/标的填充变量。
  - 多模型/多提供商按方法单独配置。

## 5. 关键产品决策
- `AI Analysis` 是独立于现有 holdings/symbol analysis 的“通用分析工作台”。
- 变量来源仅为 Prompt 中出现的 `${VAR}` 占位符，前端自动提取并生成输入项。
- Settings 中的方法管理仅支持新增、编辑、删除。
- 历史记录必须持久化。
- 仅支持用户手动填写变量。
- 分析执行必须支持流式输出。

## 6. 功能需求
- FR-1：导航栏新增 `AI Analysis` 入口，支持路由跳转与激活态展示。
- FR-2：`AI Analysis` 页面展示方法列表/选择器，并可加载某个方法的变量表单。
- FR-3：系统从 `system_prompt` 和 `user_prompt` 中提取全部 `${VAR}`，去重后作为输入字段。
- FR-4：当变量缺失、方法不存在、Prompt 为空时，页面或接口给出明确错误。
- FR-5：用户提交后，后端完成占位符替换并调用 AI 模型执行分析。
- FR-6：分析执行使用 SSE 流式返回 `progress / delta / result / error / done` 事件。
- FR-7：每次执行后持久化运行记录，至少包含方法快照、变量值、渲染后 Prompt、模型、状态、结果、时间。
- FR-8：Settings 中可新增方法、编辑方法、删除方法；删除后不影响历史记录回看。
- FR-9：`AI Analysis` 页面可查看历史运行记录，并切换查看某次结果详情。

## 7. 非功能需求
- NFR-1：变量替换必须是纯文本替换，不执行任意模板表达式。
- NFR-2：历史记录需要保留方法快照，避免方法后续修改导致历史不可追溯。
- NFR-3：SSE 过程中页面刷新或中断不会破坏已持久化完成的历史数据。
- NFR-4：首版交互保持与现有 SPA 一致，不引入新的前端框架。
- NFR-5：数据库迁移需兼容现有 SQLite 数据文件。

## 8. 验收标准
- AC-1：顶级导航可进入 `#/ai-analysis` 页面。
- AC-2：Settings 中可以新增、编辑、删除分析方法，并能持久化保存。
- AC-3：方法 Prompt 中出现的 `${VAR}` 能自动生成输入项，重复变量只出现一次。
- AC-4：调用分析后页面能流式显示输出，并在完成后展示最终结果。
- AC-5：分析结果刷新后仍可从历史中查到。
- AC-6：删除分析方法不会删除既有历史记录，历史仍能展示当时快照。
- AC-7：自动化测试覆盖核心后端逻辑、API 行为和关键前端交互路径。

## 9. 假设与约束
- 沿用现有全局 AI 设置（`base_url`、`model`、`api_key`），方法本身不携带独立模型配置。
- `${VAR}` 命名采用大小写字母、数字、下划线组合；展示时直接使用变量名。
- 首版变量输入统一按文本处理。
- 历史详情至少展示变量输入、渲染后的 Prompt 和最终结果文本。
